## 2. Use Cases (UC)

### UC-01: Регистрация и вход
**Актор:** новый пользователь  
**Предусловие:** приложение установлено  

1. Открывает приложение → экран онбординга (3 слайда с ценностью продукта)
2. Нажимает «Зарегистрироваться» → вводит имя, email, пароль
3. Система создаёт аккаунт, возвращает accessToken + refreshToken
4. Токены сохраняются локально → пользователь попадает на Home

**Альтернатива:** уже есть аккаунт → «Войти»  
**Постусловие:** пользователь авторизован

---

### UC-02: Создание группы и приглашение участников
**Актор:** авторизованный пользователь  

1. На Home нажимает «+» → «Создать группу»
2. Вводит название группы, выбирает валюту
3. Группа создана → система генерирует invite-код и QR
4. Пользователь делится ссылкой с участниками

**Постусловие:** группа создана, участники могут присоединиться по ссылке

---

### UC-03: Сканирование чека через QR ФНС
**Актор:** авторизованный пользователь — член группы  
**Предусловие:** открытый счёт существует  

1. В счёте нажимает «Сканировать» → выбирает «QR-код»
2. Камера считывает QR фискального чека
3. Фронт парсит строку → запрос к API ФНС
4. Получает структурированный список позиций с ценами
5. Пользователь проверяет и подтверждает → позиции добавляются в счёт

**Постусловие:** позиции добавлены с 100% точностью  
**Альтернатива:** QR не читается → предложить OCR или ручной ввод

---

### UC-04: OCR-сканирование чека через фото
**Актор:** авторизованный пользователь  

1. Нажимает «Сканировать» → «Фото»
2. Делает фото или выбирает из галереи
3. On-device OCR распознаёт позиции и суммы
4. Экран редактирования: пользователь проверяет и исправляет ошибки
5. Подтверждает → позиции добавляются в счёт

**Постусловие:** позиции сохранены в bill  
**Ограничение на Web:** OCR недоступен, предлагается ручной ввод (kIsWeb)

---

### UC-05: Распределение позиций между участниками
**Актор:** авторизованный пользователь — член группы  

1. В счёте открывает список позиций
2. Для каждой позиции отмечает участников (один или несколько)
3. Система автоматически делит сумму позиции между участниками
4. Итоговый экран показывает: кто сколько должен

**Постусловие:** доли рассчитаны, счёт готов к оплате  
**Валидация:** сумма долей по каждой позиции = цена × количество

---

### UC-06: Рулетка — случайный выбор плательщика
**Актор:** авторизованный пользователь  
**Предусловие:** счёт открыт, участники добавлены  

1. В счёте нажимает «Крутануть рулетку»
2. Полноэкранная анимация с именами участников
3. Система генерирует победителя (SecureRandom), сохраняет результат
4. Победитель получает push «Сегодня платишь ты 🎰»
5. Все участники видят победителя в счёте

**Постусловие:** победитель зафиксирован, повторный спин невозможен

---

### UC-07: Оплата долга через реквизиты и закрытие счёта
**Актор:** победитель рулетки + участники-должники  

1. Победитель нажимает «Я заплатил» → статус счёта: PAID_BY_WINNER
2. Система создаёт транзакции для каждого должника
3. Должники видят реквизиты победителя (карта / СБП / ЮMoney) прямо в счёте
4. Каждый переводит деньги → нажимает «Оплачено»
5. Когда все погасили → счёт автоматически переходит в SETTLED

**Постусловие:** счёт закрыт, баланс группы обновлён

---

### UC-08: Push-уведомление о долге
**Актор:** система  

1. После создания транзакций система отправляет push должникам
2. Уведомление: «Ты должен [сумма] → [имя кредитора]»
3. Пользователь открывает уведомление → попадает в нужный счёт
4. Отмечает «Оплачено»

**Постусловие:** пользователь осведомлён о долге, может сразу действовать

## 3. ПМИ — Плюсы / Минусы / Интересно

### Плюсы
- **QR ФНС** — 100% точность для российских чеков, ни у одного конкурента нет
- **Встроенные реквизиты** — путь от «сколько должен» до «перевёл» в 2 касания, это реальная незакрытая боль
- **Рулетка** — вирусная механика: люди делятся результатом → органический рост без затрат
- **Flutter** — единая кодовая база iOS/Android/Web, маленькая команда справится
- **Нет платёжного процессинга** — не нужна финансовая лицензия, юридически проще и дешевле

### Минусы
- **Качество OCR** зависит от освещения, угла, формата чека — пользователи будут сталкиваться с ошибками
- **OCR не работает на Flutter Web** — нужен fallback на ручной ввод через kIsWeb
- **Firebase push** — не работают на Android-устройствах без Google Mobile Services
- **Нет GET /groups в API** — список групп кэшируется локально в Hive, при переустановке теряется
- **Холодный старт** — без базы пользователей invite-механика и рулетка не раскрываются полностью

### Интересно
- **Корпоративный сегмент** (офисные обеды, командировки) — потенциально более высокий ARPU чем у студентов, стоит проверить через CustDev
- **Момент конверсии** — после 3-го скана чека: пользователь уже почувствовал ценность, но не исчерпал лимит — оптимальный момент для paywall
- **Сезонные скины рулетки** (Новый год, 8 марта) — низкий порог покупки, высокая конверсия в праздники
- **Deeplink → СБП/Тинькофф** — не платёжный процессинг, но резко сокращает путь до реального перевода, может стать отдельным Premium-преимуществом
